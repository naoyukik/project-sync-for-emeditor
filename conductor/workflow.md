# プロジェクトワークフロー

## 基本原則

1. **Planこそが信頼できる唯一の情報源:** すべての作業は `plan.md` で追跡されなければならない。
2. **技術スタックの計画的な変更:** 技術スタックへの変更は、実装*前*に `tech-stack.md` に文書化されなければならない。
3. **テスト駆動開発 (TDD):** 機能の実装前に単体テストを記述する。
4. **高いコードカバレッジ:** すべてのモジュールにおいて、92%を超えるコードカバレッジを目指す。
5. **ユーザーエクスペリエンス第一:** すべての決定はユーザーエクスペリエンスを優先するべきである。
6. **非インタラクティブ & CI対応:** 非インタラクティブなコマンドを優先する。テストやリンターなどのウォッチモードを持つツールは、一度だけ実行されるように `CI=true` を使用する。

## フェーズ構成の標準 (Standard Phase Structure)

Conductor で生成される `plan.md` は、原則として以下の構成を維持しなければならない。

### フェーズ 1: 問題の把握と詳細設計 (Discovery & Detailed Design)
- [ ] Task: `autonomous-researcher` スキルによる詳細調査と `evidence_report.md` の作成
- [ ] Task: 調査結果に基づいた `plan.md` の以降のタスクの具体化
- [ ] Task: Conductor - User Manual Verification 'フェーズ 1: 問題の把握と詳細設計' (Protocol in workflow.md)

### 以降のフェーズ: 具体的な実装と検証 (Specific Implementation)
- フェーズ 1 で具体化された詳細タスク群（TDD に基づく実装とテスト）を配置する。
- **品質確保タスク**: 各フェーズの実装完了後、必ず以下のタスクを実行し、警告を解消すること。
  - [ ] Task: `cargo clippy` の実行と警告の解消
  - [ ] Task: `cargo fmt` の実行
- [ ] Task: Conductor - User Manual Verification '<Phase Name>' (Protocol in workflow.md)
  - フェーズ内のclippyとfmtが全て解消された後、成果物をコミットすること。

## タスクワークフロー

すべてのタスクは以下の厳密なライフサイクルに従う：

### 標準タスクワークフロー

1. **タスクの選択:** `plan.md` から次に利用可能なタスクを順番に選択する。

2. **進行中のマーク:** 作業を開始する前に、`plan.md` を編集し、タスクのステータスを `[ ]` から `[~]` に変更する。

3. **失敗するテストの作成 (Redフェーズ):**
   - 機能またはバグ修正のための新しいテストファイルを作成する。
   - タスクの期待される動作と受け入れ基準を明確に定義する単体テストを記述する。
   - **重要:** テストを実行し、予想通りに失敗することを確認する。これがTDDの「Red」フェーズである。失敗するテストができるまでは先に進まないこと。

4. **テストを通過するための実装 (Greenフェーズ):**
   - 失敗しているテストを通過させるために必要な最小限のアプリケーションコードを記述する。
   - テストスイートを再度実行し、すべてのテストが通過することを確認する。これが「Green」フェーズである。

5. **リファクタリング (任意だが推奨):**
   - テストが通過するという安全性が確保された状態で、外部の振る舞いを変えることなく、実装コードとテストコードをリファクタリングし、明瞭さの向上、重複の排除、パフォーマンスの改善を行う。
   - リファクタリング後もテストが通過することを確認するために、再度テストを実行する。

6. **カバレッジの確認:** プロジェクトで選択されたツールを使用してカバレッジレポートを実行する。
   目標: 新しいコードに対して >92% のカバレッジ。

7. **変更の文書化:** 実装が技術スタックと異なる場合：
   - 実装を**停止**する。
   - 新しい設計で `tech-stack.md` を更新する。
   - 変更の理由を説明する日付入りのメモを追加する。
   - 実装を再開する。

8. **コード変更のステージング:**
   - タスクに関連するすべてのコード変更をステージングする。（コミットはタスクごとではなく、フェーズごとに行われる）。

9. **タスクサマリーの記録:**
   - フェーズの最終コミットメッセージに含めるためのタスクサマリーを準備する。

10. **プランの更新:**
    - `plan.md` を読み込み、完了したタスクの行を見つけ、そのステータスを `[~]` から `[x]` に更新する。
    - 更新された内容を `plan.md` に書き戻す。

### フェーズ完了の検証およびチェックポイント・プロトコル

**トリガー:** このプロトコルは、`plan.md` においてフェーズを完了させるタスクが完了した直後に実行される。

1.  **プロトコル開始の通知:** フェーズが完了し、検証およびチェックポイント・プロトコルが開始されたことをユーザーに通知する。

2.  **フェーズ変更におけるテストカバレッジの確保:**
    -   **ステップ 2.1: フェーズスコープの特定:** このフェーズで変更されたファイルを特定するため、まず開始点を見つける。`plan.md` を読み、*前*のフェーズのチェックポイントのGitコミットSHAを見つける。前のチェックポイントが存在しない場合、スコープは最初のコミット以降のすべての変更となる。
    -   **ステップ 2.2: 変更されたファイルのリストアップ:** `git diff --name-only <previous_checkpoint_sha> HEAD` を実行し、このフェーズで変更されたすべてのファイルの正確なリストを取得する。
    -   **ステップ 2.3: テストの確認と作成:** 各コードファイルについて、対応するテストファイルが存在するか確認する。

3.  **プロアクティブなデバッグを伴う自動テストの実行:**
    -   テストを実行するために使用する正確なシェルコマンドを通知する。
    -   通知したコマンドを実行する。

4.  **詳細かつ実行可能な手動検証プランの提案:**
    -   検証プロセスを順を追ってユーザーに案内する、ステップバイステップのプランを生成する。

5.  **ユーザーの明示的なフィードバックの待機:**
    -   ユーザーに確認を求める：「**これはあなたの期待を満たしていますか？yesで承認するか、変更が必要な点についてフィードバックを提供してください。**」

6.  **チェックポイントコミットの作成:**
    -   すべての変更をステージングする。
    -   明確で簡潔なメッセージ（例: `conductor(checkpoint): Checkpoint end of Phase X`）でコミットを実行し、コミットメッセージの本文に蓄積されたタスクサマリーを含める。

7.  **フェーズチェックポイントSHAの取得と記録:**
    -   *作成したばかりのチェックポイントコミット*のハッシュを取得する。
    -   `plan.md` を更新し、完了したフェーズの見出しを見つけ、コミットハッシュの最初の7文字を `[checkpoint: <sha>]` の形式で追加する。

8. **プラン更新のコミット:**
    - 変更された `plan.md` ファイルをステージングする。
    - この変更を説明的なメッセージとともにコミットする。

9.  **完了の通知:** フェーズが完了し、チェックポイントが作成されたことをユーザーに通知する。

### 品質ゲート

タスクを完了とマークする前に、以下を検証する：

- [ ] すべてのテストが通過している
- [ ] コードカバレッジが要件（>92%）を満たしている
- [ ] コードがプロジェクトのコーディングスタイルガイド（`code_styleguides/` で定義）に従っている
- [ ] すべてのパブリックな関数/メソッドが文書化されている
- [ ] 型の安全性が確保されている
- [ ] リンターや静的解析のエラーがない
- [ ] 必要に応じてドキュメントが更新されている
- [ ] セキュリティ上の脆弱性が導入されていない

## 完了の定義 (Definition of Done)

タスクは以下の条件を満たしたときに完了とする：

1. すべてのコードが仕様通りに実装されている
2. 単体テストが記述され、通過している
3. コードカバレッジがプロジェクトの要件（>92%）を満たしている
4. コードが構成されたすべてのリンターおよび静的解析チェックを通過している
5. 実装に関するメモが `plan.md` に追加されている
